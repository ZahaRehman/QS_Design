# QS Design Monorepo

Monorepo with two React apps:

- `apps/admin` - Admin panel
- `apps/store` - Storefront

## Local development

Install dependencies once from the root:

```bash
npm install
```

Run each app:

```bash
npm run dev:admin
npm run dev:store
```

## Build

Build all workspace apps:

```bash
npm run build
```

## Vercel deployment (monorepo)

Create two Vercel projects and connect the same repository:

1. **Admin project**
   - Root Directory: `apps/admin`
   - Framework Preset: Vite
2. **Store project**
   - Root Directory: `apps/store`
   - Framework Preset: Vite

Both apps include `vercel.json` with SPA rewrites so client-side routes resolve to `index.html`.

## Supabase setup (admin + store)

Supabase client is configured in both apps:

- `apps/admin/src/lib/supabase.js`
- `apps/store/src/lib/supabase.js`

Required env vars per app:

```bash
VITE_SUPABASE_URL=https://YOUR_PROJECT_REF.supabase.co
VITE_SUPABASE_ANON_KEY=YOUR_SUPABASE_ANON_KEY
```

Notes:

- Frontend should only use anon/publishable key.
- Never expose `service_role` key in browser env vars.
- Example template exists in `apps/admin/.env.example` and `apps/store/.env.example`.

### Vercel env configuration

Set the same env vars in each Vercel project:

- Project 1 (`apps/admin`): `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`
- Project 2 (`apps/store`): `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`

After editing local `.env`, restart the dev server.

## Supabase project structure in this repo

Supabase local and deploy assets live under:

- `supabase/config.toml`
- `supabase/migrations/`
- `supabase/functions/admin-auth/index.ts`
- `supabase/functions/admin-catalog/index.ts`

Useful scripts from the repo root:

```bash
npm run supabase:start
npm run supabase:db:push
npm run supabase:functions:serve
npm run supabase:functions:deploy:admin-auth
npm run supabase:functions:deploy:admin-catalog
```

To serve functions locally with secrets, create `supabase/.env.local`:

```bash
SUPABASE_URL=https://YOUR_PROJECT_REF.supabase.co
SUPABASE_ANON_KEY=YOUR_SUPABASE_ANON_KEY
SUPABASE_SERVICE_ROLE_KEY=YOUR_SERVICE_ROLE_KEY
CLOUDINARY_CLOUD_NAME=YOUR_CLOUDINARY_CLOUD_NAME
CLOUDINARY_API_KEY=YOUR_CLOUDINARY_API_KEY
CLOUDINARY_API_SECRET=YOUR_CLOUDINARY_API_SECRET
CLOUDINARY_UPLOAD_FOLDER=products
```

## Admin authentication flow

Admin authentication now runs through Edge Function `admin-auth`:

- `login`: validates email/password and confirms user exists in `public.admin_users`.
- `me`: validates bearer access token and confirms admin row.
- `logout`: signs user out from Supabase Auth.

The admin UI stores the session token locally and checks `me` on refresh.

### Seed first admin (best practice)

Seed admin users from a trusted server environment using the service-role key (never from browser code).

```bash
SUPABASE_URL="https://YOUR_PROJECT_REF.supabase.co" \
SUPABASE_SERVICE_ROLE_KEY="YOUR_SERVICE_ROLE_KEY" \
ADMIN_EMAIL="sadia@gmail.com" \
ADMIN_PASSWORD="SecurePassword123#" \
npm run seed:admin
```

This script creates or updates the user, ensures `app_metadata.role = 'admin'`, and upserts a row in `public.admin_users`.

## Database migration model

Migration `supabase/migrations/20260319180500_admin_auth.sql` creates:

- `public.admin_users` table (linked to `auth.users`)
- RLS policy for self-read checks
- `public.is_admin(uuid)` helper function

Migration `supabase/migrations/20260319194000_catalog_tables.sql` creates:

- `public.categories`
- `public.products`
- `public.product_categories` (many-to-many)
- `public.product_images` (Cloudinary image URLs and metadata)

## Product and category management

Admin product/category operations run through Edge Function `admin-catalog`:

- `list-categories`
- `create-category`
- `delete-category`
- `list-products`
- `create-product`
- `create-upload-signature` (for signed Cloudinary upload)

Admin UI now has separate catalog pages:

- `Categories`: create/list/delete categories
- `Products`: create products and assign existing categories
- `All Products`: view complete product details, categories, and images

Cloudinary uploads now use signed parameters generated by Edge Function:

```bash
CLOUDINARY_CLOUD_NAME=YOUR_CLOUDINARY_CLOUD_NAME
CLOUDINARY_API_KEY=YOUR_CLOUDINARY_API_KEY
CLOUDINARY_API_SECRET=YOUR_CLOUDINARY_API_SECRET
CLOUDINARY_UPLOAD_FOLDER=products
```

Admin UI requests signature from `admin-catalog`, uploads directly to Cloudinary using signed params, then stores returned URL/public ID in `product_images`.
Each product supports up to 10 images.

Security reminder:

- Keep `SUPABASE_SERVICE_ROLE_KEY` server-side only.
- Do not place service-role keys in any `VITE_*` variable.
